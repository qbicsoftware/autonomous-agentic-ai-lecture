Agent = LLM + Loop + Tools

  • An autonomous system that reasons, plans, and acts via external tools

How Agents Talk to LLMs

  • System Prompt primes the model with identity, rules, and tool descriptions
  • Tool Schemas are injected into the prompt so the LLM "knows" what's available
  • Function Calling API is trained to generate valid JSON tool calls matching a schema
  • The LLM probabilistically generates which tool to call — it is not a hard-coded rule
{ "name": "run_python", "description": "Execute Python code for data analysis", "parameters": { "type": "object", "properties": { "code": { "type": "string", "description": "Python code to run" } }, "required": ["code"] } }

MCP — Model Context Protocol

  • An open standard (pioneered by Anthropic) for connecting agents to external systems
  • Think of it as USB-C for AI capabilities
  • An MCP Server exposes resources, prompts, and tools
  • Any MCP Client (agent) can consume them dynamically

Why This Revolutionizes Coding Agents

  • Dynamic discovery: Agents adapt to your stack instead of relying on hardcoded scripts
  • Universal plug-ins: One protocol connects any data source or tool
  • For researchers: automate analysis, literature search, version control, and experiments
  • Decoupled architecture: Tool authors and agent builders work independently

Inference Providers: The Abstraction Layer

  • Inference providers run LLMs behind a standardized API
  • Examples: OpenAI API, Anthropic API, Groq, Together, Ollama, vLLM
  • They expose the same interface: chat completions with function calling
  • You swap providers by changing a base URL and API key
  • ⚠️ Caveat: Function calling capability varies across providers and models — local models may have limited or inconsistent tool support
POST /v1/chat/completions { "model": "gpt-4o" | "claude-3-opus" | "llama-3.1-405b", "messages": [...], "tools": [...] // Function schemas go here }

The OpenAI API Standard

  • Not OpenAI-specific — it's the de facto industry standard
  • Open-source servers (Ollama, vLLM, llama.cpp) all implement it
  • Any client code using this standard works with any provider
  • Function calling, streaming, tool use — all normalized
  • ⚠️ Caveat: Function calling capability varies across providers and models — local models may have limited or inconsistent tool support

Provider-Agnostic Agents

🔌 What You Can Do

  • Use GPT-4 for reasoning
  • Use Claude for long context
  • Use Llama for on-premise privacy
  • Use Groq for speed
  • All with the same agent code

🛠️ How It Works

  • Configure base_url + api_key
  • Same JSON schema for tools
  • Same chat completion calls
  • Same response parsing

The agent doesn't care who runs the LLM — only that the protocol is followed.

Build Your Own Agent in 50 Lines

import openai # OpenAI-compatible client client = openai.OpenAI( base_url="https://api.openai.com/v1", # Or Groq, Ollama, etc. api_key="..." ) import json def agent_loop(prompt, tools): messages = [{"role": "user", "content": prompt}] while True: response = client.chat.completions.create( model="gpt-4o", messages=messages, tools=tools ) msg = response.choices[0].message if msg.tool_calls: messages.append(msg.model_dump()) # or construct manually: {"role": "assistant", "content": ..., "tool_calls": [...]} for call in msg.tool_calls: # call.function.name / call.function.arguments (parse with json.loads) result = execute_tool(call.function.name, json.loads(call.function.arguments)) messages.append({ "role": "tool", "tool_call_id": call.id, "content": str(result) }) else: return msg.content # Done!

This works with any OpenAI-compatible provider.
